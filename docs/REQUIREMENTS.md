# 初版验收矩阵

本文件把产品计划转换为可验证要求。最终测试前逐项填写证据，不以“页面存在”代替行为验证。

## 用户端

- [ ] 底部提供首页、中央“+ / 我也吃过”、地图、我的入口，并适配移动端安全区。
- [ ] 首页顶部包含头像/用户入口和搜索栏，支持品类、地点二级筛选。
- [ ] 推荐主体仅为菜品或套餐；卡片展示图片、名称、商家、评分、价格和个性化理由。
- [ ] 菜品详情展示评价，收藏操作收藏所属商家。
- [ ] 游客可浏览、搜索、地图和收藏；评价会跳转登录并保留草稿。
- [ ] 评价支持 1–5 星、文字和最多 9 张图片，并显示机器/人工审核状态。
- [ ] 地图支持价格、类别、口味、收藏筛选；普通点、收藏星标和含收藏商家的聚合星标正确。
- [ ] “我的”展示已发布评价数、累计阅读次数、收藏、评价和退出登录。
- [ ] 用户名或邮箱登录、邮箱验证、密码找回路径存在；第三方登录仅保留扩展接口。
- [ ] PWA 可安装，公开壳可离线打开，退出后不缓存个人数据。

## 管理端

- [ ] 管理 Web 使用独立端口和登录态。
- [ ] 用户可查询、冻结和恢复，不能查看或修改明文密码。
- [ ] 商家及其菜品/套餐可新增、编辑、上下架和地图选点。
- [ ] 校园地点、品类和标签不是前端硬编码，并支持 CSV 导入校验。
- [ ] 机器审核未通过或不确定的评价进入人工队列，可通过、驳回或下架。
- [ ] 管理员变更写入可查询的审计日志。

## 后端与 AI

- [ ] FastAPI 按认证、目录、发现、地图、评价、收藏、上传、个人中心和管理端拆分路由。
- [ ] 所有校园业务资源包含并强制校验 `campus_id`。
- [ ] 商家评分只由已发布菜品评价按计划中的贝叶斯/数量权重聚合。
- [ ] 游客收藏和行为在登录时幂等合并。
- [ ] DeepSeek 只能重排数据库候选 ID，超时、限流、坏 JSON 和虚构 ID 均回退确定性排序。
- [ ] 评价文本先经过规则和 DeepSeek；模型不可用时转人工而不是阻塞或误发布。
- [ ] 图片进行真实 MIME、大小、解码、重编码和 EXIF 清除校验。
- [ ] API 错误采用 Problem Details，列表采用游标分页，写入与事件支持幂等。

## 文档、质量与交付

- [ ] `/docs`、`/redoc`、`/openapi.json` 可用，并提供中文 API 使用说明。
- [ ] 后端单元/集成测试、前端组件测试和关键用户/管理流程 E2E 均通过。
- [ ] 无 DeepSeek、高德 Key 时仍可运行确定性推荐和示意地图。
- [ ] 主要阶段合并到 `main`；若出现远端或本地冲突，停止并通知用户，不强制合并。
- [ ] 最终测试开始前先通知用户。
