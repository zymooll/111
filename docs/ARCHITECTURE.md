# 架构说明

初版采用模块化单体：FastAPI 负责用户和管理 API，React PWA 负责移动用户端，独立 React Web 负责管理端。用户端、管理端和 API 默认端口分别为 `5173`、`5174`、`8000`，共享业务数据库但使用不同令牌 audience 和权限体系。

## HTTP 模块边界

公开 API 在 `backend/app/api/router.py` 聚合以下 focused routers：

- `auth.py`：游客、注册、登录、刷新、邮箱验证和密码重置。
- `discovery.py`：推荐流、搜索建议和综合搜索。
- `catalog.py`：校园、地点、品类、标签、商家、菜品详情和公开评价列表。
- `map.py`：地图筛选和 GeoJSON 聚合。
- `favorites.py`：收藏写入与收藏列表。
- `reviews.py`：评价创建、编辑、删除和阅读计数。
- `profile.py`：偏好、本人评价和统计。
- `events.py`：行为事件批量写入。
- `uploads.py`：评价图片安全处理。

`interactions.py` 只保留第三方扩展导入兼容，不参与主应用路由装配。管理 API 由 `backend/app/admin/` 提供用户、目录、商家、菜品、导入、评价审核和审计能力。

## 校园隔离

校园隔离是数据层和服务层的共同约束：

- `CampusArea`、`Category`、`Tag`、`Merchant`、`MenuItem`、`Favorite`、`Review`、`ReviewView`、`InteractionEvent`、`AdminAuditLog` 和 `ImportJob` 都持久化非空 `campus_id`。
- 资源创建时验证地点、品类、标签、商家和菜品属于同一校园。
- 资源读取和修改时要求调用方声明 `campus_id`，跨校园 ID 按不存在或参数无效处理。
- 推荐、评分、画像、地图、统计、导入和审计查询均限定在校园范围内。
- 数据迁移 `20260718_0002_campus_isolation_and_idempotency.py` 为既有数据回填校园并建立约束。

首版 UI 默认选择一个校园，但后端边界按多校园隔离实现。

## 动态目录

地点、品类和标签以数据库为唯一业务来源。用户端先读取 `/campuses`，再按校园加载 `/areas`、`/categories` 和 `/tags`；首页二级筛选、地图筛选和偏好编辑直接使用服务端 ID。管理端可维护标签字典，菜品只能从同校园字典中选择标签。

Mock 数据仅属于显式演示适配器，不作为真实 API 的目录别名或回退 ID。

## 推荐与画像

推荐主体始终是菜品或套餐，商家只作为归属和收藏实体。推荐流程：

1. 数据库召回当前校园中已上架的合法候选。
2. 使用评分、评价量、预算、避忌、显式口味、常去地点和收藏执行确定性排序。
3. `services/recommendations.py` 提供确定性排序和本地推荐理由。
4. 可选 DeepSeek 只重排数据库候选 ID，并为候选生成简短理由。
5. 模型未配置、超时、限流、JSON 无效、ID 重复或出现候选外 ID 时返回确定性结果。

行为画像只使用点击、收藏、详情阅读和经本地目录匹配后的搜索信号。曝光用于分析但不反向推断偏好，避免推荐自我强化。发送给 DeepSeek 的画像不含账号、邮箱、评价原文和原始搜索文本。

## 评价、评分与图片

评价先执行本地规则，再调用 DeepSeek 审核；模型不可用或不确定时进入人工队列。含图片评价进入人工确认。只有 `published` 且未删除的评价参与公开列表和评分：

- 菜品评分使用校园已发布评价均值作为先验的贝叶斯平滑。
- 商家评分按菜品评价数量平方根加权，避免少量评价菜品支配结果。
- 状态变化、编辑和删除后立即重算相关菜品评分。

图片上传执行真实格式识别、大小限制、完整解码、像素限制和重新编码。评价只能引用本人上传目录中的现存图片，重编码时不复用上传 EXIF。

## 游标与幂等

业务列表采用不透明游标：时间有序资源使用 `services/pagination.py` 的时间戳加资源 ID 游标；推荐流可使用内部偏移表示，但对客户端仍是不透明 `cursor`。路由层不使用数据库 offset 分页。

写入幂等分为两层：

- HTTP 层：`services/idempotency.py` 接收 `Idempotency-Key`，保存成功 JSON 响应；相同请求重放结果，不同请求复用同一键返回 `409`。
- 领域层：收藏使用唯一约束，评价阅读和行为事件使用稳定 `event_id`，游客合并按目标账号去重。

幂等记录只保存请求摘要和响应，不保存 Bearer Token 明文。

## 错误与可观测性

请求中间件生成或透传 `X-Request-ID`。Starlette/FastAPI HTTP 异常、请求校验、响应校验以及未处理异常统一转换为 Problem Details。内部异常只向客户端返回稳定中文描述，详细堆栈写入服务端日志。

## 地图与坐标

商家保存 WGS-84 原始坐标和 GCJ-02 展示坐标。字段名明确标识坐标系，地图 GeoJSON 额外返回顶层 `coordinate_system`。

- 后端在低缩放级别按 Web-Mercator 像素网格聚合，聚合点携带 `has_favorite`。
- 用户端配置高德 Key 后使用高德 JS API 和 MarkerCluster，并显示收藏商家或含收藏成员聚合点的星标。
- 无高德 Key 或加载失败时使用校园示意地图，保留相同筛选、聚合和收藏语义。
- 管理端地图选点输出 WGS-84；首版不内置商业坐标转换服务，生产导入数据应同时提供准确 GCJ-02 坐标。

## PWA 与缓存边界

用户端通过 Workbox 缓存公开应用壳和静态资源，API 个人响应不做持久化缓存。登录状态下收藏仅保存在内存并从服务端恢复；退出或令牌过期时清除账号派生收藏、查询缓存和评价草稿，再进入新的游客会话。匿名设备收藏可以继续作为游客数据保留。

## 部署边界

SQLite 开发模式不依赖独立 Worker。Compose 提供 PostgreSQL/PostGIS、Redis、Mailpit 和 API 联调环境；Redis/PostGIS 当前为后续缓存、限流、任务和空间索引升级预留，不是核心浏览流程的硬依赖。
