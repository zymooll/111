# 架构说明

初版采用模块化单体。HTTP 请求由 FastAPI 按鉴权、发现、目录、评价、地图、个人中心和管理端拆分；DeepSeek 调用、评分聚合和图片校验在明确超时及降级边界内完成。Redis 配置保留给后续缓存、限流和异步任务，当前 SQLite 开发模式不依赖独立 Worker。

## 边界

- 推荐实体始终是菜品或套餐，商家只作为归属和收藏实体。
- 评价只关联菜品或套餐，商家评分由已发布的菜品评价加权聚合。
- 所有业务数据包含 `campus_id`，首版只展示一个校园。
- 用户端与管理端使用不同前端端口、令牌 audience 和权限体系，但共享业务数据库。
- DeepSeek 只接收偏好标签等去标识化画像和数据库候选数据；故障时不得阻断核心浏览流程。
- 行为画像只使用点击、收藏、详情阅读和搜索汇总；曝光事件用于分析但不反向推断偏好，避免推荐自我强化。

## 坐标

商家保存 WGS-84 原始坐标和供高德展示的 GCJ-02 坐标。初版使用普通数值列和应用层聚合，PostgreSQL/PostGIS 容器为后续空间索引与视口查询升级预留。任何包含经纬度的 API 都必须标明 `coordinate_system`。
